# ERP-система для автоматизации работы зоопарка

## Предметная область

Программа моделирует систему управления зоопарком, которая автоматизирует:
- Учет животных разных видов (львы, попугаи, змеи)
- Проведение медицинских осмотров при поступлении
- Кормление животных по расписанию
- Очистку вольеров
- Организацию выставок с участием животных

## Реализация принципов SOLID

### Single Responsibility Principle

- Класс `Animal` отвечает только за состояние и базовое поведение животных
- Класс `AnimalService` управляет только регистрацией и поиском животных
- Класс `FeedingService` отвечает исключительно за операции кормления
- Класс `MedicalService` занимается только медицинскими осмотрами
- Класс `ReportingService` генерирует отчеты, не изменяя данные

### Open/Closed Principle

- Для добавления нового вида животного достаточно унаследовать класс от `Animal` без изменения существующего кода
- Новые типы сотрудников добавляются через наследование от `Employee`
- Интерфейсы сервисов позволяют расширять функциональность без модификации клиентского кода

### Liskov Substitution Principle

- Классы `Lion`, `Parrot`, `Snake` могут использоваться везде, где ожидается `Animal`
- Классы `Zookeeper`, `Veterinarian` корректно заменяют `Employee`
- Все наследники полностью реализуют контракты родительских классов и не нарушают их поведение

### Interface Segregation Principle

- `FeedingService` содержит только методы, связанные с кормлением
- `MedicalService` включает исключительно медицинские операции
- `ExhibitionService` отвечает только за управление выставками
- `EnclosureService` содержит методы для работы с вольерами
- Каждый интерфейс имеет минимальный и специфичный набор методов

### Dependency Inversion Principle

- Сервисы зависят от абстракций репозиториев (`AnimalRepository`, `EnclosureRepository`), а не от конкретных реализаций
- Внедрение зависимостей происходит через конструкторы
- Возможна замена реализаций репозиториев без изменения бизнес-логики

## Потенциальные проблемы при расширении

1. **Жесткая привязка логики кормления** - инструкции по кормлению жестко заданы в классах животных, что усложнит введение сезонных или возрастных диет

2. **Управление сложными связями** - при добавлении функционала учета совместимости животных могут потребоваться дополнительные сервисы для избежания циклических зависимостей

3. **Масштабирование системы хранения** - переход от in-memory хранилищ к базам данных потребует адаптации репозиториев, хотя архитектура это предусматривает

4. **Валидация бизнес-правил** - добавление сложных правил проверки совместимости животных на выставках может потребовать введения дополнительных сервисов валидации

## Преимущества введенных абстракций

1. **Расширяемость** - возможность легко добавлять новые виды животных и типы сотрудников без изменения существующего кода

2. **Тестируемость** - зависимость от интерфейсов позволяет использовать mock-объекты для юнит-тестирования

3. **Низкая связанность** - разделение ответственности уменьшает влияние изменений в одном модуле на другие

4. **Поддерживаемость** - четкая структура и соблюдение принципов SOLID делают код понятным и предсказуемым

5. **Заменяемость компонентов** - возможность замены реализаций репозиториев без изменения бизнес-логики

6. **Повторное использование кода** - абстрактные классы и интерфейсы способствуют повторному использованию через наследование и композицию

## Компиляция и запуск

### Требования

- Java Development Kit (JDK) 8 или выше
- Командная строка (терминал)

### Компиляция

```bash
# Переход в директорию проекта
cd /путь/к/проекту

# Компиляция всех Java файлов
javac -d out src/**/*.java
```

### Запуск

```bash
java -cp out ZooSystem
```

### Альтернативный способ компиляции (для Windows)

```bash
mkdir out

javac -d out -sourcepath src src/ZooSystem.java
```
