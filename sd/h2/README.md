# Модуль «Учет финансов» ТигрБанка

## Предметная область

Программа моделирует базовый функционал личного финансового учета. Пользователь может создавать банковские счета, категории доходов и расходов, а также операции (доходы/расходы), привязанные к конкретному счету и категории. На основе операций автоматически рассчитывается баланс счета. Система позволяет анализировать доходы и расходы за период, группировать их по категориям, выполнять импорт и экспорт данных, а также пересчитывать баланс при необходимости.

Основные сущности:

- **BankAccount** – счет: идентификатор, название, баланс.
- **Category** – категория: идентификатор, название, тип (доход/расход).
- **Operation** – операция: идентификатор, тип (доход/расход), ссылка на счет, сумма, дата, описание, ссылка на категорию.

## Соответствие принципам SOLID

1. **Single Responsibility Principle**
    - Каждый класс отвечает за одну четко определенную зону ответственности. Например, `BankAccount` хранит только данные счета, `BankAccountService` содержит бизнес-логику работы со счетами, а `BankAccountRepository` отвечает за доступ к данным.
    - Отдельные интерфейсы и реализации для работы с сущностями (`*EntityService`) отделяют операции с БД от бизнес-логики.

2. **Open/Closed Principle**
    - Введение интерфейсов (`BankAccountService`, `AnalyticsService`, `Exporter`, `Importer`) позволяет расширять функциональность без изменения существующего кода. Например, добавление нового формата экспорта требует лишь создания класса, реализующего `Exporter`, и регистрации его в контейнере.

3. **Liskov Substitution Principle**
    - Все реализации интерфейсов взаимозаменяемы. Например, `CsvExporter`, `JsonExporter`, `YamlExporter`, `PdfExporter` могут использоваться там, где требуется `Exporter`, без изменения клиентского кода (`ExportService`).

4. **Interface Segregation Principle**
    - Интерфейсы небольшие и специфичные. Например, `BalanceCalculator` содержит только метод расчета баланса, `BalanceRecalculationService` – методы пересчета. Клиенты не зависят от методов, которые они не используют.

5. **Dependency Inversion Principle**
    - Модули верхнего уровня (например, `DemoRunner`, `ExportService`) зависят от абстракций (интерфейсов), а не от конкретных реализаций. Все зависимости внедряются через конструктор (DI).

## Использование DI-контейнера

В проекте активно используется Spring Framework как DI-контейнер. Все бины (сервисы, репозитории, мапперы, экспортеры, импортеры) управляются контейнером. Внедрение зависимостей выполняется через конструктор (благодаря Lombok `@RequiredArgsConstructor`), что упрощает тестирование и повышает гибкость.

Пример: `ExportService` получает список всех реализаций `Exporter` через конструктор, что автоматически собирает все зарегистрированные в контейнере экспортеры.

## Проблемы при добавлении нового функционала и расширении бизнес-логики

В текущей архитектуре при добавлении новой сущности (например, «Бюджет» или «Цель») потребуется:
- Создать новую JPA-сущность, репозиторий, DTO, маппер.
- Реализовать сервисный слой (EntityService для работы с БД и бизнес-сервис).
- Расширить функционал экспорта/импорта: модифицировать интерфейсы `Exporter` и `Importer` или создать новые версии, что потребует изменений во всех существующих реализациях. Это может быть решено путем введения универсального контейнера данных или версионирования форматов.

Другая потенциальная проблема – усложнение логики пересчета баланса при введении новых типов операций (например, переводы между счетами). Сейчас пересчет опирается на `OperationTypeEnum` с функцией эффекта. При добавлении нового типа, меняющего баланс иначе, потребуется изменить этот enum и, возможно, логику калькулятора. Это можно смягчить, сделав эффект частью метаданных типа операции (например, хранить в БД правило расчета).

## Аргументы в пользу введенных абстракций

- **Абстракция `Exporter`/`Importer`** позволила легко добавить поддержку четырех форматов без изменения сервиса экспорта. Каждый формат инкапсулирует свою логику сериализации, клиентский код не зависит от деталей.
- **Разделение на `*EntityService` и бизнес-сервисы** (например, `BankAccountEntityService` и `BankAccountService`) четко отделяет операции с БД от бизнес-логики, упрощает тестирование и повторное использование.
- **Использование интерфейсов для аналитики (`AnalyticsService`)** позволяет в будущем заменить реализацию (например, на более производительную или с кэшированием) без изменения клиентов.
- **Мапперы (MapStruct)** абстрагируют преобразование между сущностями и DTO, уменьшая дублирование кода и повышая читаемость.

## Работа с данными из файлов

Реализован импорт и экспорт данных в форматах CSV, JSON, YAML (PDF только экспорт).
- **Экспорт** – все счета, категории и операции сохраняются в файл выбранного формата.
- **Импорт** – данные читаются из файла, и если записи с такими ID еще не существуют, они создаются в БД.
- Каждый формат реализован отдельным классом, что соответствует принципу открытости/закрытости.

## Статистика и пересчет баланса

- Для измерения времени выполнения методов используется AOP-аспект `ExecutionTimeAspect` с аннотацией `@Measured`. Это позволяет централизованно добавлять логирование времени без изменения бизнес-логики.
- Пересчет баланса выполняется автоматически на основе всех операций счета через `BalanceRecalculationServiceImpl`. Предусмотрен также ручной пересчет с коррекцией баланса, если автоматическое значение не совпадает с ожидаемым.

## Единый стиль кода

В проекте соблюдается стандартный стиль Java (Oracle Code Conventions). Код отформатирован единообразно, используются соглашения об именовании классов, методов, переменных. Для упрощения кода применяется Lombok.
